# RFC-0002: Privacy Cascade Protocol

- **Author(s):** SuperInstance (@superinstance)
- **Status:** Proposed
- **Created:** 2026-02-10
- **Updated:** 2026-03-25
- **Supersedes:** None
- **Superseded by:** None

## Abstract

This RFC specifies the Privacy Cascade Protocol — a multi-tier escalation mechanism that allows mask-lock chips to collaborate across device boundaries while maintaining mathematically provable privacy guarantees. The protocol defines three escalation levels (Local, Edge, Cloud), data transformation requirements at each boundary, and the audit trail format.

## Motivation

A single mask-lock chip has limited compute capacity. A USB-C dongle with an 8×8 array can run small models, but a 1B-parameter transformer requires partitioning across multiple chips. The challenge: how do chips share intermediate activations without leaking private information?

Current approaches are inadequate:
- **Federated learning:** Designed for training, not inference; gradient sharing still leaks information
- **Secure enclaves (SGX/TrustZone):** Software-based, vulnerable to side-channel attacks
- **Homomorphic encryption:** 10,000× performance overhead, impractical for real-time inference

The Privacy Cascade Protocol provides hardware-enforced privacy boundaries with minimal performance overhead (<5% latency increase per escalation level).

## Specification

### 1. Cascade Levels

```
┌─────────────────────────────────────────────────────────┐
│  Level 0: LOCAL                                         │
│  ┌─────────┐                                            │
│  │ Chip    │  Raw activations, full precision            │
│  │ (on-die)│  No data leaves the chip                    │
│  └─────────┘  Privacy guarantee: absolute (hardware)     │
├─────────────────────────────────────────────────────────┤
│  Level 1: EDGE                                          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│  │ Chip 0  │◄──►│ Chip 1  │◄──►│ Chip 2  │             │
│  └─────────┘    └─────────┘    └─────────┘             │
│  Anonymized features only                               │
│  XOR masking with per-session key                       │
│  Privacy guarantee: computational (AES-128 equivalent)  │
├─────────────────────────────────────────────────────────┤
│  Level 2: CLOUD                                         │
│  ┌─────────┐         ┌─────────────┐                   │
│  │ Edge    │────────►│ Cloud       │                   │
│  │ cluster │◄────────│ aggregator  │                   │
│  └─────────┘         └─────────────┘                   │
│  Differential privacy (ε=1.0, δ=1e-5)                  │
│  Aggregated statistics only, no individual data          │
│  Privacy guarantee: mathematical (DP composition)       │
└─────────────────────────────────────────────────────────┘
```

### 2. Level 0 → Level 1 Transition (Local to Edge)

#### 2.1 Trigger

Escalation occurs when:
- Host writes `CMD = CASCADE_ESCALATE` (0x04) to command register
- Current `PRIVACY_LEVEL` is 0
- Cascade target chip ID is set in `CASCADE_CTRL` register

#### 2.2 Data Transformation

Before activations leave the chip, they are transformed:

```
1. Feature extraction:
   raw_activations[N] → features[M]  where M << N
   (e.g., 768-dim hidden state → 64-dim feature vector)

2. XOR masking:
   session_key = TRNG(128)  // hardware true random number generator
   masked_features = features XOR expand(session_key, M)

3. Quantize:
   masked_features_q = quantize_int8(masked_features)
```

**Normative:**
- Feature extraction MUST reduce dimensionality by at least 4×
- Session key MUST be generated by hardware TRNG (not PRNG)
- Session key MUST be rotated every inference cycle
- Original raw activations MUST be discarded after feature extraction (not cached)

#### 2.3 Cascade Packet Format

```
┌────────────────────────────────────────────────┐
│ Cascade Header (16 bytes)                      │
├────────┬───────┬──────────┬───────────────────┤
│ src_id │ dst_id│ layer_id │ privacy_level     │
│ [31:0] │[31:0] │ [15:0]   │ [1:0] + reserved  │
├────────┴───────┴──────────┴───────────────────┤
│ Payload (variable)                             │
│ masked_features_q[0..M-1]                     │
├────────────────────────────────────────────────┤
│ Integrity (32 bytes)                           │
│ HMAC-SHA256(header + payload, session_key)     │
└────────────────────────────────────────────────┘
```

#### 2.4 Inter-Chip Transport

| Form Factor | Transport | Bandwidth | Latency |
|-------------|-----------|-----------|---------|
| Thunderbolt box | PCIe Gen3 x4 | 3.9 GB/s | <1 μs |
| UCIe chiplet | UCIe 1.0 x16 | 64 GB/s | <10 ns |
| Battery pack | UART | 1 MB/s | ~1 ms |
| Genesys2 cascade | UART | 1 MB/s | ~1 ms |

### 3. Level 1 → Level 2 Transition (Edge to Cloud)

#### 3.1 Trigger

Escalation occurs when:
- Host writes `PRIVACY_LEVEL = 2` to privacy register
- Level 1 cascade is already active
- Cloud endpoint is configured in extended registers

#### 3.2 Differential Privacy

Before any data leaves the edge cluster:

```python
def apply_differential_privacy(features, epsilon=1.0, delta=1e-5):
    """Apply Gaussian mechanism for (ε,δ)-differential privacy."""
    sensitivity = compute_l2_sensitivity(features)
    sigma = sensitivity * sqrt(2 * ln(1.25 / delta)) / epsilon
    noise = gaussian(0, sigma, shape=features.shape)
    return features + noise
```

**Normative:**
- Default privacy budget: ε=1.0, δ=1e-5
- MUST use Gaussian mechanism (not Laplace) for L2-sensitive data
- Privacy budget MUST be tracked across queries (composition theorem)
- When budget is exhausted, cloud escalation MUST be refused
- Budget resets every 24 hours (configurable via extended registers)

#### 3.3 Aggregation Requirements

- Cloud MUST receive aggregated results from ≥3 edge devices (k-anonymity)
- Individual device contributions MUST NOT be recoverable
- Cloud responses are public (no privacy requirement on return path)

### 4. De-escalation

De-escalation is immediate and unconditional:

| Transition | Trigger | Latency |
|-----------|---------|---------|
| Level 2 → Level 1 | Write `PRIVACY_LEVEL = 1` | 1 clock cycle |
| Level 1 → Level 0 | Write `PRIVACY_LEVEL = 0` | 1 clock cycle |
| Level 2 → Level 0 | Write `PRIVACY_LEVEL = 0` | 1 clock cycle |

**Normative:**
- De-escalation MUST clear all cascade buffers
- Session keys MUST be destroyed on de-escalation
- In-flight cascade packets MUST be dropped (not delivered)

### 5. Audit Trail

Every escalation event is logged in an append-only audit register:

```
┌─────────────────────────────────────────┐
│ Audit Entry (16 bytes)                  │
├──────────┬────────┬────────┬───────────┤
│ timestamp│ event  │ from   │ to        │
│ [63:0]   │ [7:0]  │ [1:0]  │ [1:0]     │
│ (cycles) │        │ level  │ level     │
├──────────┴────────┴────────┴───────────┤
│ metadata (4 bytes)                      │
│ [31:16] dst_chip_id                    │
│ [15:0]  layer_id                       │
└─────────────────────────────────────────┘
```

**Event types:**
| Code | Event |
|------|-------|
| 0x01 | ESCALATE |
| 0x02 | DE_ESCALATE |
| 0x03 | BUDGET_WARNING (80% consumed) |
| 0x04 | BUDGET_EXHAUSTED |
| 0x05 | PACKET_DROPPED |
| 0x06 | INTEGRITY_FAILURE |

**Normative:**
- Audit log MUST be append-only (hardware-enforced, no clear command)
- Audit log MUST survive power cycles (stored in non-volatile register file)
- Audit log capacity: minimum 1024 entries per chip
- Overflow: oldest entry is overwritten (circular buffer with overflow counter)

## Rationale

### Why three levels?

Two levels (local vs remote) is too coarse. The edge tier provides a middle ground where chips can collaborate without cloud connectivity — critical for:
- Air-gapped environments (military, medical, industrial)
- Bandwidth-constrained deployments
- Latency-sensitive applications

Four or more levels add complexity without clear benefit. The three-tier model maps naturally to trust boundaries: on-chip (absolute trust), same-device/rack (high trust), internet (low trust).

### Why XOR masking instead of encryption?

Full AES encryption at wire speed requires dedicated hardware (AES-NI equivalent). XOR masking with a TRNG-generated key provides:
- Computational security equivalent to OTP for single-use keys
- Near-zero latency (combinational logic only)
- Minimal area overhead (~200 gates)

The key insight: session keys are single-use and never transmitted, so XOR masking IS a one-time pad.

### Why ε=1.0?

The differential privacy parameter ε=1.0 balances utility and privacy:
- ε=0.1: Very private but significant accuracy loss (~30%)
- ε=1.0: Strong privacy with moderate accuracy impact (~10%)
- ε=10.0: Weak privacy, minimal accuracy impact (~1%)

Medical and financial applications may tighten to ε=0.5. General consumer applications may relax to ε=2.0. The default of 1.0 is a widely accepted standard in the DP literature.

## Backwards Compatibility

This RFC extends MLS v1.0 (RFC-0001) without breaking changes:

- New registers use offsets 0x20-0x2F (reserved range in RFC-0001)
- New commands use codes 0x04-0x07 (within allowed extension range)
- Chips that do not implement cascade simply ignore cascade commands (status: not_supported)

## Security Considerations

### Threat Model
- **Honest-but-curious edge chips:** May try to learn from received activations
  - Mitigation: XOR masking ensures activations are indistinguishable from random
- **Malicious cloud:** May try to reconstruct individual inputs from aggregates
  - Mitigation: Differential privacy with composition budget tracking
- **Network attacker:** May intercept cascade packets
  - Mitigation: HMAC integrity + XOR confidentiality
- **Physical attacker on edge chip:** May probe cascade bus
  - Mitigation: Session keys stored in volatile registers only

### Known Limitations
- Side-channel on cascade bus timing could leak inference structure (which layers cascade)
- Mitigation: constant-time cascade mode (pad all packets to max size)
- Privacy budget is per-chip; colluding chips could combine budgets
- Mitigation: global budget manager in multi-chip deployments

## Unresolved Questions

1. **Multi-tenant cascade:** Should multiple independent inference sessions share a cascade bus? Current spec assumes single-tenant.
2. **Cascade authentication:** Should chips authenticate each other before cascade? Current spec relies on physical trust (same PCB/enclosure).
3. **Budget federation:** How should privacy budgets be managed across organizational boundaries?
